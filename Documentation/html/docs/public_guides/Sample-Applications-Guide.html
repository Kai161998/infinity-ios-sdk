Sample Applications Guide Document

The sample applications that ship with the Analytics and Optimize SDKs are designed to give you, the developer, real-world examples of how to use the libraries.

There are two sample applications, both identical in functionality. To demonstrate how to use the SDK in your application, we have written the sample app in both Objective C and Swift. The app shows off how to use both the Analytics and Optimize parts of the SDK, as well as how to use an Apple Watch with your app.

Setting Up the Sample App

The sample applications can be found on the Webtrends Mobile SDK Github repository. Within this repo you will find the iOSSampleApps folder, which contains both versions of the app.

  1. Download the application to your local system. You don’t need the entire repo, but you may clone it to get the content you need.
  2. Ensure you have Cocoapods installed on your system. For more information on installing and using Cocoapods, see the Getting Started Guide for details.
  3. Install the SDK by running pod install from a terminal window in the application of your choice (Swift or Objective C).
  4. Open the .xcworkspace file file (not the SwiftSample.xcodeproj file). This file is created when the cocoapod is installed.
  5. Compile and run the app on a simulator or a physical device. Choose the ObjSample or SwiftSample target and select a simulator.

A Walkthrough of the App

Open the application and click the Sign In button. It is not necessary to use a login ID or password. These fields are just dummies.

This is the main menu page. From here, you can navigate to anywhere in the app using the buttons on the main view or the tab bar buttons along the bottom. Click the Transactions button.

Here you can type something in the search bar and tap enter. This will trigger a dummy search method. The method does not actually search for anything, but it does trigger a popup to indicate that a search event has occurred. The search method makes a call to the SDK to trigger a new Analytics search event.

Peeking at the code in the TransactionsViewController class shows exactly what is going on here:

func searchBarSearchButtonClicked(searchBar: UISearchBar) {
    // Create an event for our product view
    // Start by creating a meta object containing basic information about the event
    // eventPath can be nil if you want to use the default eventPath.
    // customParams are an optional dictionary containing string keys and string values.
    let meta = WTEventMeta(eventPath: nil, description: "user triggered search", type: "tap", customParams: nil)

    WTDataCollector.sharedCollector().triggerEventForSearchPhrase(searchBar.text, searchResult: "Search result", eventMeta: meta)

    self.view.endEditing(false)
    let alert = UIAlertView(title: "Search Event Triggered", message: "Search event has been added to data collector", delegate: nil, cancelButtonTitle: "OK")
    alert.show()
}

This event will show up in Analytics reporting as a SearchPhrase event containing the parameters above.

Next, open the Products tab, either by navigating back to Home or by clicking on the Products tab on the tab bar. This view shows three images and three buttons. By clicking on any one of the buttons, you will trigger another event, this time for a ProductId event. The relevant code (from the ProductViewController class) is shown below:

    let meta = WTEventMeta(eventPath: "/bank/products", description: "user tapped card product", type: "tap", customParams: ["myCustomParam":"myCustomValue"])

    // Once we've set up our basic information, pass it to the data collector
    WTDataCollector.sharedCollector().triggerEventForProductId(cardString, productSku: cardSku, contentGroup: "Logged In", eventMeta: meta)

The Invest tab item takes you to a page containing a UIWebView that pulls content from a web site. This is a demonstration of the hybrid (mixing native iOS and web browser) functionality of the SDK. Session information that is started in the native app can be continued onto an Analytics-equipped web page and vice versa. This makes reporting on an end user very easy, since the session remains consistent from app to web.

The Visit Our Web Site link also demonstrates hybrid functionality, but shows how a session can continue into a native Safari session and vice versa.

Next, the Settings tab takes you to a page that controls a few aspects of the SDK itself. The buttons set configuration settings in the SDK that affect its behavior. Each of these buttons toggles a boolean setting on or off. The specific call to the SDK looks like this:

let enabled = WTDataCollector.sharedCollector().valueForConfigKey(settingName) as! Bool

The settingName string is set in the Story Board and passed into this class at runtime.

Finally, the About Us page is designed to show off the features of using Optimize™ in your application. The images below show the page rendered after applying several Optimize™ tests to the UI elements on the page (left) and the design view in the storyboard in Xcode, which shows the default UI elements (right). As you can see, the background color has been changed as well as the text fields and buttons. This change is done live when the SDK initializes and is rendered when the user navigates to this page.

Each of these UI elements are Webtrends SDK-specific subclasses of standard Cocoa Touch classes. For example, the text field that contains “Optimized Text Field” is a WTOptimizeTextField subclass of UITextField. Using this subclass and setting the WtIdentifier property enables it to be a target of an Optimize test.

Clicking on the Greet button at the top demonstrates the use of a live in-the-moment (non-cached) Optimize test. Unlike normal cached tests, the content of the test is not downloaded ahead of time. When the user clicks one of the greetings, the SDK calls home to the Optimize™ server, sending along some data, downloads content returned to it based on that data, and displays the results. In the case of the greeting, the app sends greeting=personal or greeting=generic. The Optimize server has a rule that responds to the value of greeting and returns some customized text.

Clicking on Show Optimize Examples takes you to a sample page full of optimized UI elements. This page demonstrates some of the functionality available when using Optimize™ in an iOS application. If you open the application storyboard and look at this view, you will notice that the top and bottom UI elements are styled exactly the same. When rendered at runtime, the tests downloaded from the Optimize™ server are applied, changing fonts, colors, background colors, etc. For more information about customizing the UI or affecting app behavior using Optimize, see the Factor JSON Formatting Guide document.

Next Steps

Now that you’ve seen some of the things that are possible with the Webtrends iOS SDK for Analytics and Optimize™, feel free to poke around the source code to see how it all works. We encourage you to experiment with the code and try new things. The SDK is very powerful, but also very simple to use. Go have some fun with it.

You may also want to explore using the Apple Watch extension for the SDK. For more information on that, please see the WatchKit SDK Guide.
