Analytics - Webtrends WatchKit SDK Guide Document

The Webtrends SDK provides an optional library that allows developers to track events in a WatchOS extension. To install the library, follow the directions provided in the Getting Started Guide first.

Additional Swift Setup

Assuming you’ve followed all the steps in the Getting Started Guide, you’ll still need to import the Webtrends watchOS SDK to integrate the library into your Swift-based watchOS extension’s bridging header file. Note: This is a seperate bridging header file than you used for your main application. Your watchOS Extension is sandboxed from the main application, and shouldn’t run the entire core SDK, just the light-weight watchOS SDK defined in WebtrendsWatchSDK.h.

In your extension’s bridging header, simply add the following line, which will make the entire Webtrends Watch library available to your swift app:

#import "WebtrendsWatchSDK.h"

Requirements

The Webtrends watchOS SDK works by providing a thin layer for sending events to the parent app, which must be running the Webtrends SDK. This keeps battery usage to a minimum and keeps all your data in one place. The watchOS SDK also only tracks events that occur while the watch is in communication with its parent phone. Future versions of the SDK may support sending events from a queue when a connection is reestablished, but in this version, that feature hasn’t been implemented.

Additionally, in order for the parent application to see messages from the watchOS SDK, you must implement the following methods early in the application’s lifecycle (in the AppDelegate, for example).

The Webtrends watchOS SDK works by hooking in to the Watch Connectivity framework. If you havenâ€™t already set up a WCSession in both your parent application and your watchOS application, see Apple’s watchOS Watch Connectivity documentation for information on how to do so. Once you have, implement the following code in your WCSessionDelegate:

@available(iOS 9.0, *)
func session(session: WCSession, didReceiveMessage message: [String : AnyObject]) {
    let collector = WTDataCollector.sharedCollector()
    if collector.saveWatchEvent(message) {
        return
    }
    // Your code here for handling the receipt of a message from the watch.
}
- (void)session:(WCSession *)session didReceiveMessage:(NSDictionary<NSString *,id> *)message
{
    WTDataCollector *collector = [WTDataCollector sharedCollector];
    if ([collector saveWatchEvent:message]) {
        return;
    }

    // Your code here for handling the receipt of a message from the watch.

}

Your app can do whatever additional legwork it needs to do inside this method; the SDK will attach itself and log events behind the scenes.

Integrating with Your Extension

If you have automatic events (see the Automatic Events Guide) enabled in your parent app, the SDK will automatically track activation and deactivation of any WKInterface.

Alternatively, you can manually log events as follows using any of the helper functions documented in . For example, a watch button touch event could be sent to the parent app as follows:

let meta = WTWatchEventMeta(eventPath: nil, description: "MyInterfaceController", type: "Tap Event", customParams: nil)
WTWatchDataCollector.sharedCollector().sendEventForAction(meta)
WTWatchEventMeta *meta = [WTWatchEventMeta eventMetaForPath:nil description:@"MyInterfaceController" type:@"Tap Event" customParams:nil];
[[WTWatchDataCollector sharedCollector] sendEventForAction:meta];

For more information on manually tracking watchOS events, see the Manual Events Guide
